## 2. Phase v2.7.1 — Flow Controller FSM

**Status:** not_started
**Branch suggestion:** `claude/v2.7.1-flow-fsm`
**Tox rebuild:** API only
**Marquee feature of v2.7. ~1500-1800 LOC + chat UI work.**

### Why
The canonical failing prompt "Build a kaleidoscope feedback loop" (v2.4 live-debug source) wandered for 30+ turns before the agent self-diagnosed the numerical bug. Activity log + journal hints (v2.5) help the agent SEE its loops. Flow FSM goes further: it STRUCTURES the build. The agent commits to a plan, declares state transitions, backtracks explicitly on failure, exports the trace as Mermaid.

This is the "co-pilot embedded in the network" feature described in the original v2.2→v3.0 [ROADMAP.md](../ROADMAP.md) north-star vision.

### Template schema
```yaml
name: build_feedback_loop
description: Construct a feedback chain with decay tuning
version: 1
initial_state: survey
states:
  survey:
    goal: Inventory existing nodes; identify free space for chain
    suggested_tools: [td_get_info, td_get_nodes, td_locations]
    exit_when:
      kind: tool_called
      tool: td_create_node
    transitions_to: build
  build:
    goal: Create render + feedback TOPs, wire feedback chain
    suggested_tools: [td_create_node, td_connect_nodes, td_set_params]
    exit_when:
      kind: condition
      condition: feedback_chain_wired         # checked via td_get_connections post-tool
    transitions_to: tune
  tune:
    goal: Adjust decay multipliers to achieve visible accumulation
    suggested_tools: [td_set_params, td_capture_frame]
    exit_when:
      kind: tool_called
      tool: td_capture_frame
      result_filter: visual_ok                # heuristic: non-black, non-saturated
    transitions_to: snapshot
  snapshot:
    goal: Snapshot the working state
    suggested_tools: [snapshot_save_scoped]
    exit_when:
      kind: tool_called
      tool: snapshot_save_scoped
    transitions_to: complete
  complete:
    terminal: true
on_abort:
  rollback_to: snapshot                       # restore last snapshot if user aborts
```

### Tool surface
```python
@mcp.tool()
async def td_flow_start(template: str, params: dict = {}) -> dict:
    """Start a flow from a named template. Returns flow_id + initial state."""

@mcp.tool()

    """Current flow state, history, suggested next tools."""

    """Abort current flow. Runs on_abort.rollback_to if defined."""

    """Render current flow as Mermaid stateDiagram-v2."""

    """List available flow templates with descriptions."""
```

### Auto-start detection
On user prompt, runtime checks template `description` + first-state `goal` against prompt using hybrid retrieval (v2.6.1). If RRF score > threshold (e.g., 0.05), suggest flow start in chat UI (banner: "This looks like a 'build feedback loop' task. Start flow? [yes / no / never for this session]").

User can also explicitly invoke via `/flow start build_feedback_loop` slash command in chat (parsed by HTML chat panel, dispatched as `td_flow_start`).

### Persistence
Flow state stored in `comp.storage['tdpilot_api_flow_state']` so it survives `.tox` reloads (Phase 1.2.2 auto-mirror pattern). Resume on chat-pipe restart.

### Mermaid panel
- Chat HTML grows a "Flow" tab (alongside existing Chat tab)
- Lazy-loads `mermaid.min.js` from CDN (10.6+) when tab opens
- Renders `td_flow_export_mermaid()` output
- Highlights current state via Mermaid's class-based styling
- Auto-refreshes on `flow_state_change` WS event

### Persistence + cancel
- `td_flow_abort` always works — agent can exit flow on irrecoverable error
- User can click "Abort flow" in chat banner
- Aborts trigger `on_abort.rollback_to` if defined (restores named snapshot)

- Mermaid rendering — pinned strings for each of 4 templates (4 tests)
- Persistence across `.tox` reload (using `comp.storage` mock) (3 tests)

- **Risk:** FSM rigidity frustrates users. **Mitigation:** `td_flow_abort` always works; flows are opt-in (default off, must be invoked).

## 3. Phase v2.7.2 — Full `td_self_update` + Tox Updater

**Status:** not_started
**Branch suggestion:** `claude/v2.7.2-self-update`
**Tox rebuild:** Both `.tox` files (UI banner lives in API tox, version-bind in MCP tox)
**Depends on:** v2.5.7 `td_check_for_updates` (shipped in v2.5).

### Why
v2.5.7 surfaces "update available" but offers no apply-path. v2.7.2 closes the loop: server wheel auto-swap, `.tox` source-hash-gated swap, snapshot of pre-update state, clear messaging on rebuild-required paths.

Architecturally fraught (per v2.5 audit): `.tox` files are TD-baked binaries; server can self-update wheels but cannot rebuild `.tox` outside TD. The mitigation is **never auto-apply without user click + source-hash check**.

### Pre-flight checks
```bash
git tag --list "v2.5.7*"                                       # td_check_for_updates exists (or rolled into v2.5.0)
grep "td_check_for_updates" src/td_mcp/registry/               # confirmed present
```

### Files to create
- `src/td_mcp/lifecycle/self_update.py` — orchestrator: download wheel, swap-atomic, snapshot pre-state
- `src/td_mcp/lifecycle/tox_updater.py` — source-hash compare + atomic `.tox` swap (or guide to TD-rebuild)
- `tests/test_v27_self_update.py` — ~15 tests (mock GitHub API + filesystem)

### Files to modify
- `src/td_mcp/registry/lifecycle_tools.py` — register `td_self_update` (existing file from v2.5.7)
- `td_component/tdpilot_api_runtime.py` — yellow "Update available" banner on chat-pipe start when `check_for_updates` reports `has_update=True`
- `td_component/tdpilot_api_chat.html` — banner UI with "Preview" / "Apply" / "Dismiss"
- `td_component/tdpilot_api_web_callbacks.py` — POST `/self-update/apply` endpoint (requires re-auth even in `Authmode=open` — destructive op)

### Tool surface
```python
@mcp.tool()
async def td_self_update(
    component: Literal["server", "tox", "all"] = "all",
    dry_run: bool = False,
    confirm_token: Optional[str] = None,        # user-provided; prevents agent-driven auto-apply
) -> dict:
    """Apply available updates. server=swap Python wheel. tox=swap .tox if source-hash matches.
    Requires confirm_token (echoed from check_for_updates output) — prevents drive-by application."""
```

### Server upgrade flow
1. `td_check_for_updates()` returns latest tag + URL + a `confirm_token` (HMAC-signed by server-side secret, valid 5min)
2. Agent presents to user: "Update available: v2.6.2 → v2.7.0. Apply? Token: ..."
3. User clicks "Apply" in banner → frontend calls `/self-update/apply` with token
4. Server validates token, downloads new wheel from GitHub Release asset
5. Atomic install: `pip install --upgrade <wheel>` (or `npm i -g tdpilot-dpsk4@<version>` for npm install path)
6. Surface "restart MCP server" advice (we can't restart ourselves cleanly)

### Tox upgrade flow
1. Compute current `.tox-source-hash.json`
2. Download newer release's source-hash from Release assets
3. **If source hashes match** (no editor-rebuild needed because no source file changed between releases — patch release) → swap `.tox` binaries atomically:
   - Snapshot current `.tox` files as `.tox.pre-update-<timestamp>`
   - Replace with new `.tox`
   - Trigger `chat_web.par.autorestartpulse + .reloadsrc` (per v2.3.0 Bug 11 pattern)
4. **If source hashes differ** (minor/major release with source changes) → message:
   > "Source rebuild required. New release added/changed source files. Open TouchDesigner, run the canonical rebuild snippet from `feedback_td_tox_rebuild_recipe`, then re-invoke `td_self_update`."

### Pre-update snapshot
Before ANY swap (server wheel OR `.tox`):
1. Snapshot current `.tox` → `.tox.pre-update-<timestamp>` (real copy, not symlink — per v2.2.1 auto-mirror lesson)
2. Snapshot current `pyproject.toml` + `src/td_mcp/__init__.py` to a state file
3. On apply failure, restore from snapshot atomically + log

### Tests (~15)
- Mock GitHub API: latest > current, equal, older (3 tests)
- Source-hash match → atomic swap (3 tests)
- Source-hash mismatch → user-guidance message (1 test)
- Snapshot creation + rollback on failure (3 tests)
- confirm_token HMAC validation (2 tests)
- Tool registered (1 test)
- Banner UI render (mocked WS) (2 tests)

### Validation gates
```bash
uv run pytest tests/test_v27_self_update.py -v
# Manual (CAREFUL): on a test machine, run td_self_update(dry_run=True), verify advice
# Manual: NEVER test apply path on the main dev machine without backups
```

### Risks + mitigations
- **Risk:** Server-version + `.tox` drift after partial update. **Mitigation:** version-binding check on every chat-pipe start; "incompatible .tox" banner if mismatch.
- **Risk:** Agent-driven auto-apply (Bug 8 class). **Mitigation:** `confirm_token` required + Tool Approval gate (v2.5.3) routes through user click.
- **Risk:** Download interrupted mid-swap. **Mitigation:** download to `<file>.partial`, atomic rename only on completion.

```python
    """List connected MCP servers + their tiers + enabled state."""

    """Invoke a tool on a connected MCP server. Respects tier + allowlist."""

@mcp.tool()

    """Register a new MCP server in config. Requires Tool Approval (v2.5.3)."""
```

### Tier semantics
- `read-only`: agent can invoke tools whose name suggests read-only (heuristic: starts with `get_`, `list_`, `info`, `inspect`, `query`, `search`). Other tools require Tool Approval per-call.
- `read-write`: agent can invoke any tool in `tools_allowlist`. Tools outside allowlist require Tool Approval per-call.
- `full`: any tool. Discouraged. Requires explicit user setup.

### Composition with Tool Approval (v2.5.3)
External MCP tool calls flow through the same approval gate. Two layers:
1. Tier + allowlist check (this phase)
2. Tool Approval click-through if destructive heuristic fires (v2.5.3)
